# Refactoring Assessment

Date: 2026-05-31
Target: mycheet

## Conclusion

현재 코드베이스는 작고 기능 경계가 비교적 분명해서, 기능 축소 없이 simple is best 철학으로 보면 지금 당장 전면 리팩토링이 필요하지는 않다.

대신 복잡도가 몰리는 지점은 분명하다. 지금 필요한 것은 구조를 다시 짜는 큰 리팩토링이 아니라, 다음 변경 때 같이 가져갈 수 있는 작은 정리들이다. 특히 백엔드의 `app.go`와 프런트의 `Settings.jsx`만 관리하면 된다.

## Why This Is The Right Level

- `main.go`는 앱 부트스트랩과 윈도우/트레이 생성만 담당해서 단순하다.
- `frontend/src/App.jsx`는 모드 분기만 수행하므로 충분히 단순하다.
- `frontend/src/components/CommandPalette.jsx`와 `frontend/src/components/PostitViewer.jsx`는 책임이 좁고 읽기 쉽다.
- `frontend/src/lib/theme.js`는 길지만 대부분 순수 함수와 테마 데이터라서, 추상화를 더 늘리는 것보다 현재 형태가 읽기 쉽다.

즉, 전체 구조가 엉킨 상태는 아니다. 복잡도는 소수 파일에 집중되어 있고, 그마저도 아직은 국소 정리로 통제 가능한 수준이다.

## Where Complexity Is Concentrated

### 1. `app.go`에 책임이 많이 모여 있다

`app.go`는 아래 책임을 한 파일에서 같이 처리한다.

- 설정 로드/저장
- 포스트잇 CRUD
- 윈도우 생성/토글
- 트레이 동작
- 전역 핫키 등록/해제
- 최근 사용 정렬
- 파일 감시 준비 코드
- 경로 마이그레이션

이 정도 응집은 작은 앱에서는 흔하고, 지금 즉시 파일을 여러 개로 쪼개야 할 정도는 아니다. 다만 변경 이유가 서로 다른 코드가 한 파일에 섞여 있어서, 다음 기능이 추가되면 이 파일이 가장 먼저 유지보수 비용을 키울 가능성이 높다.

추가로 아래 항목은 현재 기준으로는 죽은 경로에 가깝다.

- `startWatcher()`
- `watchFile()`
- `file-changed-externally` 이벤트 발행
- `BrowseFile()`
- `InitWindowsAndTray()`

코드에 존재하지만 실제 프런트 사용처가 보이지 않거나 호출 흐름이 연결되지 않는다. simple is best 관점에서는 이런 코드는 "언젠가 쓸 수도 있는 준비 코드"보다 불필요한 인지 부하가 더 크다. 유지할 거면 연결하고, 당분간 안 쓸 거면 제거 후보로 보는 편이 낫다.

### 2. `Settings.jsx`가 프런트 복잡도의 중심이다

`Settings.jsx` 하나에서 다음을 동시에 처리한다.

- 기본 폴더 조회/변경
- 파일 목록 조회
- 포스트잇 연결/삭제
- 핫키 녹화와 저장
- 자동 시작 설정
- 테마 조회/변경
- 좌측 네비게이션과 페이지 전환 UI

현재 파일이 당장 읽기 불가능한 수준은 아니지만, 프런트에서 가장 먼저 분리 후보가 되는 곳은 여기다. 특히 상태와 이벤트가 많아서 이후 기능이 더 붙으면 빠르게 비대해질 가능성이 있다.

중복도도 여기서 가장 먼저 보인다.

- 경로에서 파일명을 추출하는 로직이 여러 번 반복된다.
- 파일 목록과 포스트잇 목록을 합쳐 `rows`를 만드는 로직이 렌더링 본문 안에 있다.
- `Events.On(...)` 등록은 보이지만 해제 흐름은 드러나지 않는다.

이 중 첫 두 개는 바로 정리 가능한 작은 중복이고, 마지막 항목은 현재 윈도우 생명주기에서는 문제 없을 수 있지만 리마운트가 늘어나면 중복 등록 위험이 생긴다.

### 3. `theme.js`는 길지만 리팩토링 우선순위가 낮다

테마 파일은 데이터가 많아서 길어 보이지만, 실제로는 다음 두 가지가 대부분이다.

- 테마 seed 배열
- 색상 계산용 순수 함수

이런 종류의 길이는 복잡도보다 데이터량에 가깝다. 지금 여기서 공통화나 메타 프로그래밍을 더 넣으면 오히려 단순성이 떨어질 가능성이 크다. 새 테마 군이 더 늘어날 때만 데이터 파일 분리를 고민하면 충분하다.

## Duplication Assessment

현재 중복은 "전면 리팩토링이 필요할 정도의 구조적 중복"은 아니다. 다만 다음 두 종류는 정리 가치가 있다.

추가로 현재 워크스페이스에는 `*_test.go` 테스트 파일이 보이지 않는다. 즉, 구조를 크게 흔드는 리팩토링을 하기에는 안전망이 약한 편이다.

### Small duplication worth removing

- 경로에서 파일명만 꺼내는 문자열 처리
- Wails 윈도우 참조를 꺼내고 잠금을 푸는 반복 패턴
- 선택 상태 초기화와 reload 호출처럼 함께 움직이는 UI 상태 갱신 코드

이런 중복은 공용 헬퍼나 작은 내부 함수 추출로 충분히 해결 가능하다.

### Duplication that is acceptable

- 테마 seed 데이터 반복
- 윈도우 생성 옵션의 유사한 선언
- 명시적으로 풀어쓴 UI 마크업

이 부분은 억지 추상화보다 현재 코드가 더 단순하다.

## Recommendation

### Not needed now

- 백엔드 구조를 서비스 레이어로 다시 나누는 전면 리팩토링
- 프런트 상태를 전역 스토어로 옮기는 작업
- 테마 시스템의 추가 추상화
- 작은 Wails 윈도우 설정까지 공용 팩토리로 묶는 작업

이런 변경은 현재 규모에서 얻는 것보다 잃는 것이 많다.

### Worth doing incrementally

1. `Settings.jsx` 안의 작은 중복부터 줄인다.
2. 경로 처리용 공용 헬퍼를 프런트에 하나 둔다.
3. `rows` 조합 로직을 별도 함수로 빼서 렌더링 본문을 줄인다.
4. 실제로 쓰지 않는 watcher/API 경로는 제거하거나 연결 여부를 확정한다.
5. 다음 기능이 settings에 더 붙는 시점에 `KeymapPage`를 별도 컴포넌트로 분리한다.

## Refactor trigger

아래 조건 중 하나가 생기면 그때는 리팩토링을 시작하는 편이 맞다.

- `app.go`에 또 다른 책임이 추가된다.
- `Settings.jsx`에 페이지 하나 이상이 더 붙는다.
- 이벤트 리스너가 더 늘어난다.
- 포스트잇 개수가 커져서 목록 조합 성능이 체감되기 시작한다.
- 테스트 없이 리팩토링하기 불안할 정도로 동작 간 결합이 증가한다.

## Safe order if refactoring is done later

리팩토링을 한다면 순서는 크게 잡지 말고 아래처럼 가는 편이 안전하다.

1. `parseHotkey`, `parseKey`, 경로 마이그레이션, 최근순 정렬 같은 순수/준순수 로직부터 테스트를 추가한다.
2. `Settings.jsx`의 중복 문자열 처리와 파생 데이터 계산을 작은 헬퍼로 분리한다.
3. 그 다음에만 `app.go`를 책임 기준으로 파일 분리한다. 예: config, windows, hotkeys.

테스트가 없는 상태에서 먼저 큰 구조 변경을 하는 것은 이 프로젝트에서는 simple 하지 않다.

## Final Verdict

지금의 답은 다음과 같다.

- 전면 리팩토링: 불필요
- 국소 정리: 필요
- 우선순위: `Settings.jsx` > `app.go`의 미사용/준비 코드 > 나머지

즉, 현재 코드는 아직 "리팩토링이 늦었다" 단계는 아니다. 다만 앞으로도 단순함을 유지하려면 큰 구조 개편보다 작은 중복 제거와 죽은 코드 정리를 먼저 하는 것이 맞다.
